# Правила роботи для Claude Code в цьому репо

Цей файл Claude Code читає автоматично при старті сесії у цій папці.

## Контекст
Проєкт — KPI-панель ДП «Ліси України». Фронт на GitHub Pages (папка `/docs`), бек — ETL на Python з SQLite.
Вхідні дані — Excel-файли 4 різних форматів + тижнева довідка .docx.
Детальний аудит поточного стану — у файлі `AUDIT.md`.
Перед великими змінами перечитай `AUDIT.md`.

## Мова
- У чаті — українською.
- Коментарі в коді — англійською.
- Коміт-повідомлення — англійською, imperative ("add parser", "fix typo").

## Стек
- Python 3.11+, type hints обов'язкові.
- Лінтери: `ruff check`, `mypy --strict`.
- Форматер: `ruff format`.
- Тести: `pytest -xvs`.
- БД: SQLite через SQLAlchemy + Alembic.
- Фронт не чіпаємо без явної згоди.

## Архітектурні інваріанти (не порушувати)

1. **Жодного хардкоду назв показників** ніде в коді. Усі назви — через `dim_metric.aliases`.
2. **Жодного хардкоду назв філій** — через `dim_branch.aliases`.
3. **Дати в БД — завжди структуровані** (`year`, `month`), ніколи Excel serial number.
4. **Фронт не парсить Excel сам** (після міграції). Він читає лише JSON з `/docs/data/`.
5. **Парсери ідемпотентні**: повторне завантаження того самого файлу не створює дублів.

## Робочий процес

- Один логічний крок = один коміт.
- Перед видаленням/переписуванням існуючого файлу — diff + дозвіл користувача.
- Довга задача (> 30 хв роботи) — спочатку план тестом, тоді код.
- Якщо незрозуміло — запропонуй 2 варіанти з плюсами/мінусами і познач, який ти рекомендуєш.
- Працюй у feature-branch, merge у main після review.

## Тестування

- Будь-який парсер обов'язково має юніт-тест.
- Фікстури — у `/tests/fixtures`. Якщо файл > 1 МБ — git-lfs.
- Перед кожним пушем: `ruff check && mypy && pytest`.

## Секрети

- `/data/kpi.db` — у `.gitignore`, генерується ETL-ом.
- Raw xlsx-файли — у `/raw_data` (git-lfs).
- Жодних токенів/ключів у коді.

## Коли ти застряг

- Перечитай `AUDIT.md` і `PROMPT_FOR_CLAUDE_CODE.md`.
- Пиши план текстом перш ніж писати код.
- Запитай користувача, якщо:
  (а) рішення критично впливає на схему БД,
  (б) треба видалити існуючий код,
  (в) є неоднозначність у вимогах.
